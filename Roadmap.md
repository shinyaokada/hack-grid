# ハッキング風パズルゲーム ver1 開発ロードマップ（〜ステージ2-4）

> 前提: [Overview.md](Overview.md)（ゲーム設計）・[Screens.md](Screens.md)（画面設計）・[Stages.md](Stages.md)（ステージ1-1〜2-4の詳細設計）を実装対象とする。
> Overview.md の「マイルストーン」表を実装フェーズとして具体化したもの。**ver1のゴールはステージ2-4（3段チェーン）までが通しでプレイできる状態。**

## ゴール定義（ver1 = Done の基準）

- [ ] タイトル→ホーム→ストーリー→CLI→鍵番号フォーム→クリア演出、の全画面遷移が実機で動く
- [ ] ステージ 1-1〜1-4、2-1〜2-4 の8本がすべて Stages.md の攻略フロー通りにクリア可能
- [ ] 新ステージ追加が「JSONファイル1個の追加」だけで完結する（エンジン側コード変更が不要なことを2回以上実証済み）
- [ ] クリア状態・初回ヒント既読状態がリロード後も保持される
- [ ] Stages.md 末尾の「横断チェックリスト」が全項目クリア

---

## 技術選定（前提として置く仮決定）

Overview.md で確定しているのは「Next.js」「クライアントサイドJSでJSONを解釈」「ステージデータはJSON」のみ。以下は本ロードマップ進行のための仮決定であり、Phase 0 の最初に確定させる。

| 項目 | 仮決定 | 理由 |
|---|---|---|
| フレームワーク | Next.js (App Router) + TypeScript | Overview.md 記載どおり。型でステージJSON構造を強制できる |
| 状態管理 | React State + Context（ゲーム状態は単純なので外部ライブラリ不要で開始し、肥大化したら再検討） | 依存を増やさない |
| スタイリング | Tailwind CSS | ターミナル風UIを素早く組める |
| ステージ検証 | Zod でJSONスキーマバリデーション | 「JSON1個追加するだけ」を安全にするため、壊れたJSONを早期検知 |
| テスト | Vitest（エンジンのユニットテスト）＋ 手動プレイテスト（画面） | エンジンのロジックはコマンド入出力が明確でテストしやすい |
| デプロイ | 未定（Vercel想定） | ver1完成後に判断でよい |

## フェーズ一覧（依存順）

```
Phase 0 基盤構築
   ↓
Phase 1 エンジンコア（ロジックのみ、UIなし）── Overview Phase1相当
   ↓
Phase 2 CLI画面UI（1-2が実機でクリアできる）
   ↓
Phase 3 パターン1の残スキン（1-3, 1-4）── Overview Phase2相当（スキン差し替え検証）
   ↓
Phase 4 Chainパターンのエンジン拡張 + 2-1 ── Overview Phase3相当
   ↓
Phase 5 Chainスキン展開（2-2, 2-3）+ 3段チェーン（2-4）── Overview Phase4の一部相当
   ↓
Phase 6 周辺画面（タイトル/ホーム/ストーリー/鍵番号フォーム/クリア演出/ガイド）
   ↓
Phase 7 永続化（進捗・初回ヒント）
   ↓
Phase 8 通しQA
   ↓
Phase 9 ver1確定
```

Phase 1〜5 は「ロジックが先、見た目は最小限」の順で進める。CLI画面の骨格だけ Phase 2 で先に作ってしまい、以降のパターン追加はそのUIに乗せていく（Phase 6でようやくタイトル〜ホームなどの外側の画面を作る）。理由: コアループ（`ls`→`inspect`→`run`→`read`）が早期に触れる状態を作ることが、パズル設計そのものの検証（詰まらないか、拒否メッセージで理解できるか）に直結するため。
