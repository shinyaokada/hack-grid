# ハッキング風パズルゲーム 画面設計書 v1.2

> 前提: [Overview.md](Overview.md) のゲーム設計（コマンド仕様・パズルパターン）に準拠する。
> 「常時ステータスバーを持たない」は全画面で厳守する。
> **「自由入力ゼロ」は v1.2 で撤回した（詳細は下記変更点）**。ただし Overview.md の設計憲法が
> 本当に守りたかったこと ―― コマンドセットを固定する／`guess` のような総当たりを許さない／
> `sudo` 的万能昇格を作らない ―― は変わらず維持する。自由入力はあくまで「決まったコマンド＋
> 引数を打鍵で入力する」ことだけに使い、ゲームロジック側の制約（権限・パス制限・鍵番号は専用フォーム
> でのみ判定）は一切緩めていない。

> v1.1 変更点: 5コマンドを一括で教える専用チュートリアル画面を廃止。
> 「大枠のチュートリアル＝ステージ1-1を実際にプレイ＋ホーム画面ガイド」と
> 「各コマンドの説明＝各ステージで初出時にインラインヒント」の二段構成に変更。
>
> v1.2 変更点: CLI画面の入力方式を「コマンド→対象→引数をタップして選ぶ」方式から、
> **実際のターミナルのようにコマンドをキーボードで入力する**方式に変更。誤入力（未知の
> コマンド・引数不足）は拒否メッセージと同じ体裁でログに表示し、詰みには繋がらない。
> 代わりに画面右（狭幅では下）に**コマンド一覧ドキュメントパネル**を常設し、構文・説明・
> そのステージで初出のコマンドへのヒントを参照できるようにした。
> コマンド入力欄は出力ログと同じ枠内（1つの連続したターミナル）に統合し、鍵番号入力パネルは
> CLI画面に入った時点から常設（別画面への遷移をやめた）。
>
> **v1.2の副作用と修正**: タップ選択式だった頃は `/root/note.txt` のような root 専有ファイルも
> 「既知のパス」として選択候補に自動で出ていたため、プレイヤーは正確なファイル名を知らなくても
> 詰まらなかった。キーボード直打ちに変えたことで、`/root` は `cd`/`ls` できないため
> **正確なファイルパスをストーリー画面の文章以外から知る手段が失われていた**（1-2以降の
> 全ステージに共通する抜け）。
>
> 最初はストーリー画面のテキストに直接パスを書く案で対応したが、「ナレーターが答えを教える」
> のはハッキングゲームとしてリアリティに欠けるとの指摘を受け、**ゲーム内で発見させる方式に
> 変更した**。具体的には、各ステージの `ls` で見える範囲に guest が `read` できる「メモ」
> （引き継ぎメモ・室内メモ等、ステージの文脈に応じたフレーバー付き）を1つ置き、その中に
> 正確なパスを書く。ストーリー画面はこれまで通り曖昧な示唆（「root しか読めないファイルに
> あるらしい」）に留め、プレイヤーは `ls` → `read memo.txt` という**ゲーム内の調査行動**で
> パスを発見する（Stages.md 各ステージの「ファイルシステム」「攻略フロー」節を参照）。

## 2. チュートリアル（大枠）

### 設計方針
一括で5コマンドを教える専用画面は持たない。かわりに以下の二段構成にする。

1. **大枠のチュートリアル** = ステージ1-1を実際にプレイしてもらい、クリア直後に一度だけホーム画面の使い方を説明する
2. **各コマンドの説明** = 各ステージでそのコマンドが初めて必要になったタイミングで、CLI画面上にインラインヒントを出す（→ 2-3、5章）

### 2-1. ステージ1-1（大枠チュートリアルの実体）

使用するコマンドは `ls` `cd` `read` の3つのみ。`inspect` `run` `connect` はまだ使わせない。

**シナリオ**: `reception-pc` に `guest` としてログイン。受付に置き忘れられたメモを確認する。

| Step | 状況 | ヒント表示 | 操作 | 結果 |
|---|---|---|---|---|
| 1 | ログイン直後、何もわからない状態 | 「まずは `ls` で今いる場所を確認しよう」 | `ls` を選択 | `memo.txt` `staff/` が表示される |
| 2 | `staff/` が気になる（任意・本筋ではない寄り道） | ヒントなし | `cd` → `staff` を選択 | 拒否:「staff のみ入室可（あなた: guest）」→ 「壁がある」体験を先に刷り込む |
| 3 | `memo.txt` が読めそう | 「`read` で中身を確認しよう」 | `read` → `memo.txt` を選択 | 中身が出力される |
| 4 | 数字が判明 | 「見つけた数字を回答フォームに入力しよう」 | 鍵番号フォームに入力 | クリア |

**memo.txt の中身（例）**
```
--- memo.txt ---
今日の合言葉は 7008
（本当に大事な金庫の番号は root にしか
  見られない場所にあるらしい……）
```

末尾の一文は次ステージ（1-2、旧「ステージ1」＝`inspect`/`run` を初めて使う回）への伏線として機能させる。

**クリア演出（1-1用）**
- 気づき:「拒否されても、まず `ls` と `read` だけで進められる場所はある」
- ボタン:「つづける」→ ホーム画面ガイドへ

### 2-2. ホーム画面ガイド（ステージ1-1クリア直後のみ）

ステージ1-1クリア後、ホーム画面に遷移した際に一度だけ表示するオーバーレイ形式のガイド。以後は自動表示しない。

| 対象 | ハイライト内容 |
|---|---|
| ステージ一覧 | 「次はここから選んで挑戦しよう」 |
| クリア状態アイコン | 「クリアすると記録が残るよ」 |
| 進捗サマリ | 「全部で ◯ ステージあるよ」 |

- 「わかった」ボタンで閉じる
- スキップ不可にする必要はない（タップで即閉じてよい）

### 2-3. 各ステージ内チュートリアル（新コマンド初回ヒント）

CLI画面（4-2）の一部として、そのステージで**新しく使うことになるコマンド**が最初に登場するとき、コマンド選択欄の該当ボタンに軽いヒントを添える。

| コマンド | 想定初出ステージ | ヒント例 |
|---|---|---|
| `ls` `cd` `read` | ステージ1-1 | 2-1参照 |
| `inspect` `run` | ステージ1-2 | 「`inspect` でツールの仕様を見てみよう」「読めないなら、ツールに代わりに読ませてみよう」 |
| `status` | 早い段階の任意ステージ | 「今の自分の状態をまとめて見たいときは `status`」 |
| `connect` | パターン4系の初出ステージ | 「この端末からは弱そうな別端末に `connect` できそうだ」 |
| `back` | 詰みかけた最初のタイミング（条件発火） | 「一手戻したいときは `back`」 |
| `help` | 常時使用可能。専用ヒントは出さない（いつでも参照できるため） | — |

表示形式・消去条件は5章「新コマンド初回ヒント」に定義する。

## 4. ステージ画面

ステージ画面は3つの子画面（ストーリー表示 → CLI → 鍵番号入力フォーム）と、クリア演出で構成される。ステージ1-1も含め、全ステージがこの骨格を共有する。

### 4-1. ストーリー表示画面

#### 表示要素
- ステージタイトル
- 導入ストーリーテキスト（状況設定・侵入対象のホスト名・自分の権限・何を暴きたいのかを匂わせる）
- 「はじめる」ボタン

#### モックアップ
```
──────────────────────────
  1-2 「受付端末・裏側」

  受付のメモにあった一文が気になる。
  金庫の本当の番号は、root しか
  読めないファイルの中にあるらしい。

              [ はじめる ]
──────────────────────────
```

#### 遷移
- 「はじめる」→ CLI画面

---

### 4-2. CLI画面（ゲームプレイ本体）

#### 設計方針（重要・v1.2で変更）
プレイヤーは実際のターミナルと同じように、1行の入力欄にコマンドをキーボードで直接タイプして Enter で確定する（`run printer /root/note.txt` のように、コマンド名と引数をスペース区切りで自分で打つ）。入力したコマンド行はそのまま出力ログに追記される。

コマンド自体の**種類は固定**（8種）であり、`write` のような任意内容の自由入力や、鍵番号のブルートフォース的な入力は存在しない（鍵番号の入力は 4-3 の専用フォームに限定）。自由入力はあくまで「決まった構文をキーボードで打つ」ことのみに使う。

未知のコマンド名・引数不足など構文として成立しない入力は、権限拒否と同じ体裁のインラインメッセージとしてログに残るだけで、ゲーム状態には一切影響しない（詰みにはならない）。

#### 画面構成
```
┌───────────────────────────────┬───────────────────┐
│ 出力ログ（スクロール領域）              │ コマンド一覧ドキュメント │
│                                 │                   │
│ $ ls                           │ ls                │
│ memo.txt  printer               │  現在地の一覧       │
│                                 │                   │
│ $ read /root/note.txt              │ cd <dir>          │
│ 拒否: root のみ読み取り可（あなた: guest）│  移動する           │
│                                 │                   │
│ $ inspect printer               │ read <file>       │
│ 所有者: root / 実行可: 全員         │  中身を読む         │
│ 動作: run printer <ファイル> で表示    │                   │
│                                 │ inspect <obj> NEW │
│                                 │  仕様を調べる        │
│                                 │  ヒント: ...        │
│                                 │                   │
│                                 │ run <tool> [引数] NEW│
│                                 │  実行する           │
│                                 │  ヒント: ...        │
│                                 │                   │
│                                 │ status / back / help│
├───────────────────────────────┴───────────────────┤
│ $ _（コマンド入力欄。↑/↓キーで入力履歴を呼び出せる）        │
└─────────────────────────────────────────────────┘
```

ドキュメントパネルは常設で、狭い画面幅ではログ・入力欄の下に縦積みになる。

#### 操作フロー（`run` の例）
1. 入力欄に `run printer /root/note.txt` とタイプして Enter
2. 出力ログに `$ run printer /root/note.txt` として追記され、実行結果が続く
3. ツール名・パスの候補は選択式ではなくなったため、プレイヤーは直前の `ls`/`inspect` の出力や、ドキュメントパネルの構文表記（`run <tool> [引数]`）を見ながら自分で入力する

#### 補助要素
- `help` 入力時: 現在使えるコマンド一覧と一言説明をログに出力（別画面には遷移しない。常設のドキュメントパネルと内容は重複するが、ログに残る形でも参照できるようにする）
- `back` 入力時: 直前の1手を取り消し、対応するログ行を灰色（取り消し済み）表示に変える
- 拒否メッセージ・構文エラーはいずれもログ内にインラインで表示（Overview.md 記載の書式に統一）
  ```
  拒否: root のみ入室可（あなた: guest）
  コマンドが見つかりません: raed（help で一覧を確認できます）
  ```
- 常時ステータスバーは置かない。現在地・権限などのまとまった情報が必要な場合のみ `status` を入力して都度ログに出力する
- 新コマンド初回ヒント（2-3）はコマンド選択ボタンではなく、ドキュメントパネル内の該当コマンド行に NEW バッジ＋ヒント文として表示する（該当コマンドを一度でも入力したら、そのステージ内・以降のステージでは表示しない）

#### 遷移
- ゴールとなる情報（鍵番号など）を `read`/`run` 経由で入手すると、ログ末尾に「回答する」ボタンが出現 → 鍵番号入力フォームへ

---

### 4-3. 鍵番号入力フォーム

#### 位置づけ
CLI操作そのものは選択式（自由入力ゼロ）だが、**ステージのクリア条件の最終回答**はプレイヤーが実際に見つけた番号を入力する行為そのものが本人の理解確認になるため、この画面に限り数値入力を許可する例外とする。

#### 表示要素
- 「見つけた番号を入力してください」等の案内文
- 数値入力欄（テンキー形式を想定。桁数はステージのJSON定義に依存）
- 「回答する」ボタン
- 誤答時: 「番号が違います。もう一度確認しよう」のようなヒントレスな短いエラー表示（正解への直接的なヒントは出さない）＋ CLI画面へ戻る導線

#### モックアップ
```
──────────────────────────
  金庫の暗証番号を入力

        [ 4 ][ 9 ][ 2 ][ 1 ]

          [ 回答する ]

  ※ CLI画面に戻って確認する場合は
    「戻る」を選択
──────────────────────────
```

#### 判定
- 入力値がステージ定義の `goal` と一致 → クリア演出画面
- 不一致 → エラー表示、再入力可（回数制限なし）

#### 遷移
- 正答 → クリア演出画面
- 「戻る」→ CLI画面（ログはそのまま保持）

---

### 4-4. クリア演出画面

#### 表示要素
- 「クリア」演出（テキストアニメーション想定）
- そのステージの「気づき」テキスト（Overview.md のサンプル盤面にある一言まとめ。例:「自分は読めないが、ツールに代行させたら読めた」）
- 実行したコマンド手数（任意表示。スコアリングはしないが、手数の短さを本人が振り返れる情報として）
- ボタン: 「次のステージへ」「ホームへ戻る」

#### 遷移
- 「次のステージへ」→ 次ステージのストーリー画面（ロックされていれば表示しない）
- 「ホームへ戻る」→ ホーム画面
- ステージ1-1クリア時のみ、「つづける」→ ホーム画面ガイド（2-2）

## 5. 共通UI要素

| 要素 | 仕様 |
|---|---|
| プロンプト表記 | 常に `$` のみ。ステージ・端末が変わっても記号は変えない |
| 拒否メッセージ | 都度ログにインライン表示。ポップアップ・モーダルにしない（Overview.mdの「常時ステータスバーを持たない」思想を踏襲し、割り込みUIを増やさない） |
| ステータス確認 | 専用バーを持たず、`status` コマンドの出力としてのみログに表示 |
| 入力の受け付け方 | 権限的に無意味な組み合わせも入力自体は常に受け付け、実行後に拒否メッセージで返す（「拒否される体験」自体がゲーム性のため）。UI側で事前に入力を塞がない |
| 新コマンド初回ヒント | そのステージで初めて使えるようになったコマンドに、ドキュメントパネル上で「NEW」バッジ＋一言ヒントを添える。該当コマンドを一度入力したら、以後そのステージ内・以降のステージでは表示しない |

## 6. ステージ1-2 詳細仕様（具体例：パターン1-A「受付端末・裏側」）

Overview.md のサンプル盤面（Lv1）に対応する画面遷移の実例。ステージ1-1の続きとして、`inspect` と `run` を初めて使わせる回。

1. **ストーリー画面**
   「受付のメモにあった一文が気になる。金庫の本当の番号は、root しか読めないファイルの中にあるらしい。」→「はじめる」
2. **CLI画面**（ドキュメントパネルに `inspect`/`run` のNEWバッジとヒントが出ている状態でスタート）
   - `ls` と入力 → `memo.txt` `printer` が見える
   - `read /root/note.txt` と入力 → 拒否される
   - `inspect printer` と入力 → 仕様が判明（root所有・全員実行可・run printer <ファイル> で表示）
   - `run printer /root/note.txt` と入力 → 内容が出力される
     ```
     --- /root/note.txt ---
     金庫の暗証番号は 4921
     ```
   - ログ末尾に「回答する」ボタン出現
3. **鍵番号入力フォーム**
   - `4921` を入力 → 「回答する」
4. **クリア演出画面**
   - 「クリア」
   - 気づき:「自分は読めないが、ツールに代行させたら読めた」
   - 「次のステージへ」/「ホームへ戻る」
