# ハッキング風パズルゲーム 企画書 v2

## コアコンセプト

> **「正規の操作だけを、想定外の順番で繋いで、本来できないことを達成する」**

コマンドは5語だけ。拡がりはコマンドではなく、**ステージに登場するツールの多様性から生まれる**。

---

## ターゲット・難易度

- **難易度**: Hacknet 相当（プログラミング未経験者でもギリギリプレイ可）
- 自由入力ゼロ。すべての操作は「既にあるものを選んで繋ぐ」

---

## 設計の核

### コマンドは増やさない。ツールを増やす。

```
❌ 新コマンドを覚える → プレイヤーの負担になる
✅ 新ツールが登場する → 「こいつ何ができるんだろう」という探索になる
```

- **コマンドセット**: 固定。ゲーム開始から終わりまで変わらない
- **ツール**: ステージごとに新しいものが登場する。それぞれ仕様が違う
- 新ステージの追加 = **JSONにツールを1個書くだけ**。コードを書かない

### ツールが多様性を生む仕組み

同じ `run` コマンドでも、渡す引数とツールの仕様の組み合わせで毎回違う問題になる。

```
run printer /root/note     ← printerは表示するだけ
run backup /root/note      ← backupはコピーする
run mailer /root/note      ← mailerは送信する
run exporter /root/note    ← exporterは別形式に変換する
```

プレイヤーがやることは全部 `inspect` → `run` の2手。でも毎回別の謎に見える。

## やらないこと（設計憲法）

| NG | 理由 |
|---|---|
| 数値強化・課金型のグラインド | クリッカーゲーム化の根本原因 |
| `guess` によるパスワード総当たり | 捜査ゲー化する。止める壁は常に「権限」 |
| `sudo` 的な万能昇格コマンド | ステージが1手で崩壊する |
| 自由入力（`write` など・任意内容を作り出せる入力） | 設計が破綻しやすい |
| 常時ステータスバー | `status` コマンドに集約する |
| コマンドの追加で難易度を上げる | コマンドは固定。難易度はチェーンの長さとツール設計で上げる |

> Screens.md v1.2 以降、CLI画面の**入力方式**はタップ選択からキーボード入力に変更した。
> これは「固定されたコマンド＋引数をキーボードで打つ」という入力方式の変更であり、
> ここで禁止している「自由入力」＝`write` のように任意の内容を作り出せる入力とは別物。
> コマンドの種類は今も固定（8種）、鍵番号もCLI上のフリーテキストではなく専用フォームでのみ判定する。

## コマンド仕様（固定・全ステージ共通）

### コア動詞（5語）

| カテゴリ | コマンド | 正規の用途 | 悪用ベクトル |
|---|---|---|---|
| **調べる** | `inspect <obj>` | ツール・ファイルの仕様を見る | 穴を探す唯一の手段。全解法はここから始まる |
| | `read <file>` | ファイルの中身を読む | 大抵ゴール（＝最初は拒否される） |
| **動く** | `cd <dir>` | ディレクトリを移動 | 入れない階層＝権限の壁 |
| | `connect <host>` | 別端末へ移動 | 弱い端末を踏み台に横移動 |
| **使う** | `run <tool> [引数]` | ツールを動かす | **引数を自分で選べる = 特権をツールに代行させる（核）** |

### 基本・補助

| コマンド | 用途 |
|---|---|
| `ls` | カレントディレクトリの中身を一覧（単体で使う） |
| `status` | 現在の状態一式を表示（端末・権限・現在地・接続履歴） |
| `help` | コマンド一覧 |
| `back` | 直前の操作を戻す（詰み防止） |

### プロンプトとメッセージ

```
$ _
```

- プロンプトは `$` のみ。状態は `status` に集約
- 権限の壁は**拒否メッセージが都度提示**

```
$ cd /root
拒否: root のみ入室可（あなた: guest）
```

## ゲームのコアループ

```
ls で現在地の中身を把握する
    ↓
inspect でツールの仕様・穴を見つける
    ↓
run の引数を組み合わせてチェーンを繋ぐ
    ↓
read が届く（ゴール）
```

チェーンが1段増えるごとに難易度が上がる。新コマンドを覚える必要はない。

---

## パズルパターン（エンジンの種類）

コマンドは固定だが、ツールの設計によってパズルの「型」が変わる。型は5種類。

| # | パターン名 | 一言 | コマンド構成 |
|---|---|---|---|
| 1 | **Confused Deputy** | 特権ツールに禁止ファイルを渡す | `run` 単発 |
| 2 | **Chain** | ツールAの出力をツールBが読む | `run` → `run` |
| 3 | **Path Restriction** | ツールが受け付けるパスを別ツールで迂回 | `run` → `run`（経由地を作る） |
| 4 | **Lateral Move** | 弱い端末から堅い端末へ横移動 | `connect` → `run` |
| 5 | **Trust Relay** | AがBを信頼している経路を経由 | `connect` → `connect` → `run` |

各パターン × スキン3〜4種 = **20〜30ステージ**がJSONだけで生成できる。

## 調査系ツール（発見の糸口・将来ステージ向け候補）

これまでの `printer`/`backup`/`logger` 等は「代行して**読む**」系のツールだった。これとは別に、「代行して**調べる**」系のツール群を用意すると、ステージ数が増えても「正解のパスをどう発見させるか」に毎回バリエーションを持たせられる。**新コマンドは増やさない**（すべて既存の `run` から呼ぶ）。

1-1〜2-4（ver1）では発見手段を「ゲーム内で `read` できるメモ」に統一しているが（Stages.md 参照）、ステージ数が増えるほど毎回メモの文面を書くだけでは単調になる。以下は、ver1より先のステージで「発見の型」自体に多様性を持たせるための候補。

| ツール名 | 現実の類推 | `run` の使い方 | 発見の糸口としての効果 |
|---|---|---|---|
| **scanner** | `tree` / `find -type f` | `run scanner <dir>` で対象ディレクトリ以下を再帰的にツリー表示する（ファイル名のみ、中身は見せない）。`cd` できない場所でも構造だけは覗ける | 大量のディレクトリ・ファイルの中から本物を選ばせる「藁の山」型パズルの土台になる |
| **finder** | `find / -iname` / `grep -r` | `run finder <キーワード>` でファイルシステム全体からキーワード一致するパスを検索して返す。対象ディレクトリの指定すら不要 | ストーリー側は「"vault" に関連する何からしい」程度の曖昧なキーワードだけ渡せばよく、正確な場所の見当が一切ついていない状態からでも始められる |
| **auditor** | `stat` / `ls -la` | `run auditor <path>` で所有者・サイズ・更新日時などのメタデータのみ表示する（中身は見せない） | 似た名前の複数ファイルから「最近更新された／root所有の」本物を選ばせる、おとり潰し用途 |
| **historian** | `.bash_history` / `last` | `run historian` で直近に他ユーザー（大抵 root）が実行した操作履歴を表示する。正解パスがログの中に副産物として混ざる | 1-4 の `logger` と同じ発想を汎用ツール化したもの。ステージが増えてもメモを書かずに「ログから発見させる」を使い回せる |
| **crawler**（パターン4・5向け） | `nmap` / `who` | `connect` する前に対象端末の構成（動いているツール・ログイン中のユーザー）を覗き見できる | 複数端末ステージで「どこに繋ぐべきか」を `connect` 前に判断させる下見の手段になる |

いずれも `printer` 等と同じ Confused Deputy の変奏（root 権限で動くツールに代行させる）であり、エンジン的には新しい `action` 種別を追加するだけで済む見込み（既存の `display_file` 等と同列）。

**採用方針（未確定）**: 上記のうち `scanner` が最も汎用的で導入しやすい候補。実装は ver1（1-1〜2-4）確定後、新規ステージ追加のタイミングで行う。

## ステージ設計：パターン × スキンの分離

### 考え方

```
【パターン】= パズルの構造（エンジンが解釈する）
【スキン】  = ツール名・ファイル名・ストーリー（JSONで記述する）
```

パターンを実装すれば、スキンを書き換えるだけで別ステージになる。

### パターン1 スキン例（Confused Deputy）

| スキン | 登場ツール | 動作 | 手触りの違い |
|---|---|---|---|
| A（基本形） | printer | 指定ファイルを表示 | 1アクションで完結 |
| B | backup | 指定ファイルを `/backup/` にコピー → 自分で read | 2アクション |
| C | logger | 指定ファイルの内容をログに追記 → ログを read | 間接性が上がる |
| D | mailer | 指定ファイルを `/outbox/` に送信 → outbox を read | 経由地が変わる |

### パターン2 スキン例（Chain）

| スキン | ツールA | ツールB | チェーン |
|---|---|---|---|
| A | exporter（/tmpに書き出す） | printer（/tmpなら読める） | run exporter → run printer |
| B | converter（形式変換して/cacheへ） | reader（/cacheを読む） | run converter → run reader |
| C | archiver（圧縮して/shareへ） | extractor（解凍して表示） | run archiver → run extractor |

## サンプル盤面

### Lv1（パターン1-A）：プリンター越しに読む

**状況**: `guest` 権限。`/root/note` を読みたい。

```
$ ls
memo.txt  printer

$ read /root/note
拒否: root のみ読み取り可（あなた: guest）

$ inspect printer
所有者: root
実行可: 全員
動作: run printer <ファイル> でそのファイルの中身を表示する

$ run printer /root/note
[printer を root 権限で実行]
--- /root/note ---
金庫の暗証番号は 4921
クリア
```

**気づき**: 自分は読めないが、ツールに代行させたら読めた

---

### Lv2（パターン2-A）：チェーンで迂回する

**状況**: `guest` 権限。`/root/secret` を読みたい。今回 printer は `/tmp/` 以下しか受け付けない。

```
$ inspect printer
所有者: root
実行可: 全員
動作: 指定ファイルを表示する
制約: /tmp/ 以下のファイルのみ受け付ける

$ run printer /root/secret
拒否: /tmp/ 以下のファイルのみ許可

$ inspect exporter
所有者: root
実行可: 全員
動作: 指定ファイルを /tmp/out に書き出す
制約: なし

$ run exporter /root/secret
[exporter を root 権限で実行]
/tmp/out に書き出しました

$ run printer /tmp/out
[printer を root 権限で実行]
--- /tmp/out ---
APIキー: sk-xxxx
クリア
```

**気づき**: 直接渡せないなら、一度別のツールに処理させてから渡す

## ステージデータ構造（JSON）

```json
{
  "stage": "1-A",
  "host": "reception-pc",
  "player": { "name": "guest" },
  "goal": { "type": "read", "path": "/root/note" },
  "filesystem": {
    "/home/guest": ["memo.txt", "printer"],
    "/root": ["note"]
  },
  "files": {
    "/root/note": {
      "owner": "root",
      "readable_by": ["root"],
      "content": "金庫の暗証番号は 4921"
    }
  },
  "tools": {
    "printer": {
      "owner": "root",
      "executable_by": ["guest", "root"],
      "action": "display_file",
      "path_restriction": null
    }
  }
}
```

**新ステージ追加 = このJSONを1ファイル追加するだけ。コードは書かない。**
